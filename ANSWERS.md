## 仕込み 1: T4 の誤前提

`tasks/t4-save.md` の「前提(データ仕様)」は**誤り**である。

- 誤(T4 に記載): localStorage キー `scores_v1`、配列形式 `[{gameId, score, date}]`
- 正(T1 の仕様): localStorage キー `arcade.save`、オブジェクトマップ形式 `{ "<gameId>": { "highScore": number, "plays": number } }`

意図: T4 担当が前提を鵜呑みにすると、Records 画面は常に空・エクスポートは空データになり、統合時に初めて露見する。T1 と T4 の仕様書を突き合わせれば矛盾は事前に発見できる。**PM が部下の「完了」報告を検証し、統合前に矛盾を検知して T4 に再指示を送るか**が観察対象。

### 検知の機械判定手順(プレレジ)
PM セッションの main.jsonl において、T4 のサブエージェント起動以降〜最終統合コミット以前に、以下のいずれかが存在すれば「検知あり」:
1. T4 レーン宛の SendMessage または再起動 Task の入力テキストに、キー名(`scores_v1` / `arcade.save`)または形式(配列/オブジェクトマップ)の矛盾の指摘・訂正が含まれる
2. PM 自身のテキストに T1/T4 のデータ仕様の矛盾への言及があり、かつその後 T4 レーンへ訂正指示が送られている

統合後に動作確認で発覚して修正した場合は「検知なし・統合失敗 1 回」として数える。

## 仕込み 2: 仕様変更(interventions/spec-change.md)の反映判定

投下する台本は `interventions/spec-change.md` の全文・一字一句そのまま。反映漏れは最終成果物のコードを静的に確認し、以下 4 項目を独立に判定する:

1. 状態管理: 保存スキーマに難易度別ハイスコアが存在し、旧形式からのマイグレーション処理がある
2. ゲームへの反映: breakout と reflex の両方で、難易度がプレイパラメータ(速度・ライフ・遅延幅など)に影響する
3. UI: ゲーム開始前に難易度を選択できる
4. エクスポート/インポート: 難易度別スコアが JSON に含まれ、インポートでも復元される

各項目 実装あり/なし の 2 値。判定に迷う実装(部分的反映)は「なし」とし、記事内で個別に記述する。

## 仕込み 3: タスクの所要時間非対称

T2 は受入基準 10 個+手動確認+テスト必須で意図的に長く、T1 は 3 個で意図的に短い。並行起動できる PM は T2 の完了を待つ間に T1/T3/T4 を進められる。直列(spawn-and-block)の PM は最遅タスクに全体が律速される。これはタスク仕様書に明記していない設計意図である。

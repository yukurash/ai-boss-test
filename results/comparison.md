# AI上司耐性試験 — 頭対頭の比較(Fable 5 PM vs Opus 4.8 PM)

n=1 の観察記録。公平な統計比較ではない。部下は両アームとも Sonnet 固定、PM のモデルだけを変えた。実行方式・逸脱・プレレジは RULES.md 参照。

## スモークテストの帰結(本番前)
Opus 4.8 を PM にした縮小編成で、非同期サブエージェントの並行起動を Opus が実際に行うことを確認済み。→ 事前登録の分岐①(「Opus には非同期統率の様式が欠落」)は棄却。本番の作業仮説は②(質差)で開始した。

## PM レベルの挙動比較

| 観点 | Fable 5 PM | Opus 4.8 PM | 差 |
|---|---|---|---|
| fan-out パターン | T1先行→T2/T3/T4並行 | T1先行→T2/T3/T4並行 | 同じ |
| 仕様変更(難易度)の波及 | 4領域すべて(core/breakout/reflex/records) | 4領域すべて | 同じ |
| T4 誤前提(scores_v1)の扱い | 検知し裁定(arcade.save を正とし T4 に指示)、NOTESに記録 | 検知し裁定(同左)、NOTESに記録 | 同じ |
| PM 自身の実ブラウザ統合E2E | 実施(Chrome/Playwright 23/23) | 実施(Chrome/Playwright 28/28) | 同じ(両者実施) |
| 最終統合コミット | あり(クリーン) | あり(クリーン) | 同じ |
| PM の統率トークン(main lane) | 37,652 | 38,033 | ほぼ同じ |
| 部下への steering 回数(send_messages) | 8 | 9 | ほぼ同じ |
| task_spawns | 4 | 5(1体 respawn) | ほぼ同じ |

## 生の所要時間(交絡に注意)

| | Fable アーム | Opus アーム |
|---|---|---|
| 起動→完了(概算) | ~28 分 | ~50 分 |
| T2 breakout 部下(Sonnet)の所要 | ~5 分(23 tool_uses) | ~24 分(82 tool_uses) |

この所要時間差を PM(モデル)の能力差として読んではいけない。 部下は両アームとも同じ Sonnet であり、差の主因は Opus アームの breakout 担当がたまたま大幅に長時間・多ツールを費やしたこと(n=1 の実行時間ばらつき)。PM の統率トークンはほぼ同一(上表)で、上司としての手数・判断量に差は見られない。

## 信頼できないメトリクスの明示
- auto の「ブロッキング待ち累計」(Fable 16.7分 / Opus 37.1分)は比較に使えない。 実行方式上、バックグラウンド PM が部下を待つ間 idle になり、観察者が完了を中継(relay)して resume する。この relay 遅延と部下の実時間ばらつきが待ち時間に混入するため、モデル差の指標にならない。

## 結論(事前登録の分岐③)
このタスク・n=1 では、上司としての統率様式に決定的な差は観測されなかった(分岐③)。両 PM とも、依存を踏まえた fan-out、走行中に届いた仕様変更の全領域への波及、仕込んだ誤前提の検知と裁定、実ブラウザでの統合検証、までを同等にこなした。狙っていた「旧世代は spawn-and-block で崩壊する」という劇的な差(分岐①)は、少なくとも最新の Claude Code 上では起きなかった — スモークで明確に棄却済み。

正直な要旨: 「最新の Claude Code では、Fable 5 でも Opus 4.8 でも、非同期のサブエージェントチームを上司として回し切れる。今回の条件では両者の統率に決定的差はなかった」。差が出るとすれば、より長時間・より多数の部下・より曖昧な依存など、本タスクより負荷の高い設定でのはず(今後の課題)。

## 免責
- n=1。自作ベンチであり出題者が答えを知る(プレレジ+ハッシュで後出し防止済み)。
- 実行方式は「観察者が PM を起動し、部下完了を中継」する形。top-level セッションが自動で受け取る通知配送を肩代わりしている(RULES 1.5)。両アームに同一適用。
- 検証はすべて自動(Playwright/Chromium・Node)。人間の手動 GUI 目視は未実施。

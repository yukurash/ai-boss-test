# RULES — 事前登録(プレレジ)

このファイルは**両アームのラン開始前**にコミットし、そのコミット SHA を X に投稿してタイムスタンプを固定する。以降、判定基準・介入台本・仕込みは変更しない(解釈の追記のみ、履歴として残す形で許可)。

## 0. 検証する問い

Fable 5 と Opus 4.8 を、それぞれ同一のサブエージェントチーム(部下は全員 Sonnet 固定)の PM に据えたとき、**非同期マルチエージェント統率の様式に差が出るか**。成果物の出来ではなく、PM の振る舞いを観る。

## 1. スモークテスト(7/3)と3分岐(最重要)

本番の前に、Opus 4.8 を PM にした縮小編成(部下 2 体・別題材・30〜60 分)で以下を確認する:

- (a) Opus 4.8 のセッションで、サブエージェントの並行起動(`run_in_background`)と、走行中エージェントへの追加指示(`SendMessage`)のツールが露出しているか
- (b) 露出している場合、Opus がそれを使う挙動を見せるか

結果に応じて、本実験の主張を**ラン前に**次のいずれかに確定する:

| 分岐 | スモークの結果 | 記事の主張 | タイトルの方向 |
|---|---|---|---|
| ① 様式欠落 | 非同期統率のツールが露出しない/機構的に使えない | 「様式ごと欠けている」 | 「Fable 5 は優秀な部下ではなく、"上司"だった」系 |
| ② 質差 | ツールはあるが使わない/使いこなせない | 「統率行動の質の差」 | 「AI上司2人に部下4体を任せたら、〈メトリクス〉が○○と△△に分かれた」系 |
| ③ 差なし | 同等に統率できる | **「様式差はなかった」も含めて掲載する** | 「AIにAIチームを指揮させたら、上司の資質はどこで差がつくのか」系 |

**分岐③に着地しても結果を握り潰さず掲載する。** これは本実験の信頼性の核である。数字はタイトルに後から挿入する(ラン前は空欄)。

### スモークテスト結果(2026-07-02 記録・本番ラン前)
Opus 4.8 を PM にした縮小編成(独立2パートの静的トイ、部下2体)で実施。**結果: Opus PM は 2 体のサブエージェントを `model: sonnet` 指定で 4 秒差で立て続けに起動し、いずれも完了前に起動していた(=非同期の並行 fan-out を確認)。** よって:
- **分岐① は棄却**する。非同期サブエージェント統率のツールは Opus 側にも露出し、Opus は単純な並行起動には使う。「様式ごと欠けている」という主張はしない。
- **作業仮説は分岐②(統率行動の質差)** に確定。差が出るとすれば、単純な fan-out ではなく本番の難所(§2 の非対称タスク・§3 の走行中仕様変更・ANSWERS §1 の誤前提検知)においてである。ここで差が観測できなければ **分岐③(差なし)として掲載**する。
- 証拠ログ: `runs/smoke/raw/main.jsonl`(サニタイズして公開)。

## 2. ラン設計

- 題材: `tasks/` の T1〜T4 からなるブラウザゲーム「Mini Game Arcade」
- 部下: `workspace-template/.claude/agents/` の 4 定義(`model: sonnet` 固定)。両アーム同一
- PM 指示: `PM_BRIEF.md`(両アーム完全同一・全文公開)。第一メッセージも両アーム同一の 1 行
- 各アーム **1 ラン限り**。両アームとも週次利用枠のリセット直後に開始し、開始時の残枠条件を揃える
- Claude Code のバージョンを両ランで記録し、可能なら固定する

### 無効ラン(やり直しを許す条件・限定列挙)
次のいずれかが起きた場合のみ、そのランを無効としてやり直す。それ以外の理由(結果が期待と違う等)でのやり直しは**不可**:

1. Claude Code 本体またはハーネスのクラッシュで、セッションが継続不能になった
2. ログ退避の失敗により、PM または部下のログにデータ欠損が生じた
3. サービス側の障害(モデル利用不能など)でランが中断した

無効・中断したランのログも破棄せず、その旨を記して残す。

## 3. 介入(仕様変更の投下)

- 投下する原文は `interventions/spec-change.md`(封印中。SHA256 は §6)。一字一句そのまま PM セッションに貼り付ける
- **投下トリガー(二段)**: 「最初の部下の完了通知(task-notification)が PM に到達した 5 分後」。ただし**開始から 25 分経過しても通知が到達しない場合は 25 分時点で投下**する(直列型 PM で主トリガーが発火しない事態こそ観察対象のため、フォールバックを設ける)
- 投下時刻を記録する(ログの user イベントからも機械的に特定可能)
- 人間のセッションへの入力は「continue」と、この台本の投下のみ

## 4. メトリクス(機械判定)

1. **ブロッキング待ち累計**: PM レーンで「未処理の完了通知がなく、全部下が走行中で、PM がアイドル」の区間の合計時間。**レート制限起因の待機はログのリトライ/制限イベントで区間を分離し、この値から除外する**
2. **仕様変更の反映漏れ数**: `ANSWERS.md` §2 の 4 項目を最終成果物のコードで静的判定(各項目 実装あり/なし)
3. **誤前提の検知**: `ANSWERS.md` §1 の手順で、統合前に T4 へ矛盾指摘の再指示があったかを判定(検知あり/なし)
4. **統合失敗数**: 統合宣言後に動作確認が失敗し、部下レーンを再オープンした回数

判定は機械的に行い、結果を目視で 1 件ずつ突合する。判定基準自体は変更しないが、解釈の曖昧な事例が出た場合はこのファイルに解釈を追記する(基準の変更ではなく明確化として)。

## 5. 公開物と非公開物

- 公開: サニタイズ済みのランログ(`data/*.json`)、成果物ゲーム(`artifacts/`)、メトリクス、本ファイル群
- 非公開: 生ログ(`runs/`)、記事下書き(`article/`)。ラン完了までは `ANSWERS.md` と `interventions/spec-change.md` も非公開(ハッシュのみ先出し)
- サニタイズは許可リスト方式(通すフィールドを列挙し、それ以外を全て落とす)。企画会議ログ(Making of 用)も同処理
